# **0. 쿠버네티스 실습 환경**

## **Minikube vs K3s 비교**

| 구분 | Minikube | K3s (by Rancher/SUSE) |
| :--- | :--- | :--- |
| **설계 철학** | 로컬 개발 및 **학습용 샌드박스** | **에지(Edge)/IoT 및 운영 환경** 최적화 |
| **바이너리 크기** | 상대적으로 큼 (Full K8s 기능 포함) | **매우 작음 (< 100MB)** |
| **자원 효율성** | 최소 2GB RAM 권장 (리소스 점유 높음) | **최소 512MB RAM (극도로 효율적)** |
| **실행 방식** | VM, Docker, bare-metal (다양한 드라이버) | 단일 바이너리 (systemd 서비스로 실행) |
| **시작 속도** | 상대적으로 느림 (수 분 소요) | **매우 빠름 (수 초 내외)** |
| **주요 장점** | 풍부한 빌트인 애드온 (Dashboard 등) | 낮은 유지보수 비용, 실 운영 환경 적용 가능 |
| **리스크(한계)** | **호스트 자원 과소비:** VM 사용 시 성능 저하 | **표준과의 차이:** 경량화를 위한 일부 기능 제거 |
| **리스크 대응** | Docker 드라이버 사용 및 애드온 비활성화 | 도입 전 필수 API 및 클라우드 호환성 사전 검증 |

## **가상화 계층에 따른 성능 영향 분석**

배포판 선택 시 가장 고려해야 할 요소는 **'가상화 계층(Virtualization Layer)'** 유무에 따른 호스트 시스템의 부하입니다.

### **가. Minikube (가상 머신 드라이버 사용 시)**
* **구조:** [하드웨어] → [호스트 OS] → **[하이퍼바이저] → [게스트 OS]** → [K8s]
* **특징:** Windows/macOS 환경에서 VirtualBox, VMware 등을 통해 실행됩니다.
* **영향:** **'이중 OS 실행'** 구조로 인해 실제 서비스 동작과 무관하게 가상화 계층 유지에만 상당량의 CPU와 RAM이 소모됩니다. 이로 인해 호스트 PC의 전체적인 응답 속도가 저하될 리스크가 있습니다.

### **나. K3s (Native 실행 시)**
* **구조:** [하드웨어] → [호스트 OS] → [K8s 프로세스]
* **특징:** 가상화 계층 없이 리눅스 호스트 OS 위에서 직접 프로세스로 실행됩니다.
* **영향:** 중간 단계(오버헤드)가 없어 하드웨어 성능을 90% 이상 그대로 활용 가능하며, 사양이 낮은 저전력 에지 기기나 로컬 환경에서 극강의 가벼움을 제공합니다.

## **노트북 사양에 따른 권장사항**

단순히 배포판을 선택하는 것을 넘어, 학생들의 기기 부하를 줄이기 위한 구체적인 가이드라인이 필요합니다.

### **가. 가상화 제거 (WSL2 또는 Linux 활용)**
* **리스크:** Windows에서 VirtualBox나 VMware를 쓰는 것은 저사양 노트북에 치명적입니다.
* **대응:** Windows 사용자라면 **WSL2(Windows Subsystem for Linux)** 내에 K3s를 설치하도록 유도합니다. 가상 머신(VM)보다 가볍고 호스트 자원을 유연하게 공유합니다.

### **나. 사양에 따른 환경 추천**

* **"내 노트북이 쌩쌩하다(RAM 16GB 이상)"** → **Minikube** (공식 표준 기능과 다양한 애드온 경험)
* **"내 노트북이 힘들어한다(RAM 8GB 이하)"** → **K3s** (최적의 성능으로 실습 중단 방지)
