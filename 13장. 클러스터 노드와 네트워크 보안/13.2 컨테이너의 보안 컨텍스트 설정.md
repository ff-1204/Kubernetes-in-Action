# 13.2 컨테이너의 보안 컨텍스트(Security Context) 설정

## 들어가기 전에: "이 컨테이너는 무엇을 할 수 있는가?"

13.1이 "노드의 벽을 허무는" 거친 옵션이었다면, **보안 컨텍스트(securityContext)** 는 좀 더 세밀하게 **"이 컨테이너가 무슨 권한으로, 무엇까지 할 수 있는지"** 를 조절하는 다이얼입니다.

보안 컨텍스트는 두 곳에 지정할 수 있습니다.

* **파드 수준** (`spec.securityContext`) — 파드 안 **모든 컨테이너의 기본값**
* **컨테이너 수준** (`spec.containers[].securityContext`) — 해당 컨테이너에만 적용 (파드 수준 값을 덮어씀)

> **비교 기준:** 아무 설정 없이 띄운 파드는 보통 **컨테이너 이미지에 지정된 사용자(대개 root)** 로 실행됩니다. 아래 옵션들은 이 "기본은 위험하다"를 하나씩 잠그는 자물쇠들입니다.

---

## 13.2.1 특정 사용자로 컨테이너 실행하기 (`runAsUser`)

컨테이너를 이미지에 박힌 사용자가 아니라 **내가 지정한 UID**로 실행합니다.

```yaml
spec:
  containers:
  - name: main
    image: alpine
    securityContext:
      runAsUser: 405        # guest 사용자 UID로 실행
```

* **비유:** 이미지가 "관리자 권한으로 들어오세요"라고 해도, **"아니, 너는 손님(guest) 명찰을 달고 들어와"** 라고 강제하는 것.
* UID(숫자)를 쓰는 이유: 이미지에 해당 **사용자 이름이 없을 수도** 있어, 이름보다 숫자 ID가 안전합니다.

---

## 13.2.2 컨테이너가 root로 실행되는 것 막기 (`runAsNonRoot`)

특정 UID를 지정하긴 번거롭고, **"어쨌든 root만 아니면 된다"** 고 할 때 사용합니다.

```yaml
    securityContext:
      runAsNonRoot: true
```

* 이 옵션이 켜지면, 이미지가 root(UID 0)로 실행되려 할 경우 **컨테이너가 아예 시작되지 못하고 거부**됩니다.
* **비유:** 클럽 입구에서 **"신분증에 'root'라고 적힌 사람은 입장 불가"** 라고 막아서는 문지기.
* **왜 중요한가:** 컨테이너가 탈옥(escape)당했을 때, root였다면 노드까지 장악당하지만 일반 사용자라면 피해를 크게 줄일 수 있습니다.

---

## 13.2.3 특권 모드로 파드 실행하기 (`privileged`)

반대로, 컨테이너에 **노드의 모든 장치와 커널 기능에 대한 전권**을 주는 옵션입니다.

```yaml
    securityContext:
      privileged: true
```

* **비유:** 일반 컨테이너가 "손님"이라면, 특권 컨테이너는 **건물의 모든 문을 여는 마스터키를 든 시설관리자**입니다. `/dev`의 모든 장치를 보고 만질 수 있죠.
* **사용처:** 노드 하드웨어를 직접 다뤄야 하는 시스템 파드(예: 일부 네트워크/스토리지 플러그인).
* **위험:** 사실상 노드를 통째로 장악할 수 있는 권한이므로 **극히 제한적으로만** 사용해야 합니다.

---

## 13.2.4 커널 기능(Capabilities) 추가·제거하기

리눅스는 root의 권한을 **잘게 쪼갠 "능력(capability)" 단위**로 관리합니다. 특권 모드라는 "마스터키" 대신, **필요한 열쇠 한두 개만** 골라 주거나 빼앗을 수 있습니다.

### 기능 추가 (`add`)

```yaml
    securityContext:
      capabilities:
        add:
        - SYS_TIME        # 시스템 시간 변경 능력만 부여
```

### 기능 제거 (`drop`)

```yaml
    securityContext:
      capabilities:
        drop:
        - CHOWN           # 파일 소유자 변경 능력을 빼앗음
```

* **비유:** "마스터키(privileged)" 대신 **"창고 열쇠 하나만(add)"** 주거나, 가진 열쇠 꾸러미에서 **"위험한 열쇠 하나를 빼는(drop)"** 방식.
* **원칙:** 최소 권한(least privilege). 꼭 필요한 기능만 `add`하고, 안 쓰는 위험한 기능은 `drop`하는 것이 가장 안전합니다.

---

## 13.2.5 컨테이너 파일시스템 쓰기 막기 (`readOnlyRootFilesystem`)

컨테이너의 **루트 파일시스템을 읽기 전용으로** 잠급니다. 공격자가 침투해도 시스템 파일을 변조하거나 악성 바이너리를 심을 수 없습니다.

```yaml
    securityContext:
      readOnlyRootFilesystem: true
    volumeMounts:
    - name: my-volume
      mountPath: /writable/data    # 쓰기가 필요한 곳만 볼륨으로 허용
```

* **비유:** 도서관 책장 전체를 **"열람만 가능, 낙서·훼손 금지"** 로 잠그되, **메모지(볼륨)** 만 따로 비치해 거기에만 적게 하는 것.
* 로그·임시 파일 등 쓰기가 필요한 경로는 **별도의 볼륨을 마운트**해서 해결합니다.

---

## 13.2.6 컨테이너가 다른 사용자로 실행될 때 볼륨 공유 (`fsGroup`, `supplementalGroups`)

여러 컨테이너가 서로 **다른 UID**로 실행되면서 **같은 볼륨**을 공유하면 권한 문제가 생깁니다. 이를 그룹 단위로 풀어줍니다.

```yaml
spec:
  securityContext:
    fsGroup: 555                 # 마운트된 볼륨 파일의 그룹 소유권
    supplementalGroups:
    - 666
    - 777
```

* **`fsGroup`** — 파드가 사용하는 **볼륨 파일의 그룹 소유자**를 이 GID로 맞춰줍니다. 덕분에 다른 UID의 컨테이너들도 그룹 권한으로 같은 파일에 접근 가능.
* **`supplementalGroups`** — 컨테이너 프로세스가 **추가로 속하게 될 그룹 목록**.
* **비유:** UID가 다른 직원들이 **같은 "프로젝트 폴더(볼륨)"** 를 함께 쓰도록, 모두를 **하나의 공용 그룹**에 넣어 권한을 통일하는 것.

---

## 결론: 한 컨테이너의 자물쇠 모음

보안 컨텍스트는 **개별 파드/컨테이너가 스스로 거는 자물쇠**입니다.

| 목표 | 옵션 |
|------|------|
| 누구로 실행? | `runAsUser`, `runAsNonRoot` |
| 권한 크기 | `privileged`, `capabilities.add/drop` |
| 파일 변조 방지 | `readOnlyRootFilesystem` |
| 볼륨 공유 권한 | `fsGroup`, `supplementalGroups` |

하지만 여기엔 한계가 있습니다 — **자물쇠를 거는 건 어디까지나 파드 작성자의 양심에 달려 있다**는 점입니다. 누군가 `privileged: true`를 마음대로 쓰면 막을 길이 없죠. 그래서 **"애초에 이런 옵션을 못 쓰게 클러스터 차원에서 강제"** 하는 장치가 필요합니다. 그것이 다음 절 13.3의 주제입니다.
