# 13.1 파드에서 호스트 노드의 네임스페이스 사용하기

## 들어가기 전에: 파드는 원래 "독방"이다

기본적으로 파드의 각 컨테이너는 **자기만의 독방(리눅스 네임스페이스)** 안에서 격리되어 살아갑니다. 자기만의 네트워크 인터페이스, 자기만의 프로세스 목록(PID), 자기만의 호스트네임을 가지죠. 옆방(노드)에서 무슨 일이 일어나는지 전혀 볼 수 없습니다.

그런데 **시스템 수준의 작업**(노드 네트워크 모니터링, 노드 프로세스 디버깅 등)을 하는 파드는 이 독방의 벽을 허물고 **노드의 자원에 직접 접근**해야 할 때가 있습니다. 이때 사용하는 것이 호스트 노드의 네임스페이스를 빌려 쓰는 옵션들입니다.

> ⚠️ **이 모든 옵션은 보안 구멍이 될 수 있습니다.** 격리를 일부러 깨는 기능이므로, 꼭 필요한 시스템 파드에만 신중하게 사용해야 합니다.

---

## 13.1.1 노드의 네트워크 네임스페이스 사용하기 (`hostNetwork`)

`hostNetwork: true`로 설정하면 파드는 **자기 독방의 네트워크 벽을 허물고 노드의 네트워크를 그대로 사용**합니다. 즉, 파드가 노드의 IP·포트·네트워크 인터페이스를 직접 보게 됩니다.

```yaml
apiVersion: v1
kind: Pod
metadata:
  name: pod-with-host-network
spec:
  hostNetwork: true          # 노드의 네트워크 네임스페이스 사용
  containers:
  - name: main
    image: alpine
    command: ["/bin/sleep", "999999"]
```

* **비유:** 보통은 아파트 각 세대(파드)가 **자기 현관과 자기 우편함**을 갖지만, `hostNetwork`는 **건물 1층 로비(노드)를 통째로 내 거실로 쓰는 것**과 같습니다.
* **실제 사용처:** 쿠버네티스 컨트롤 플레인 구성요소(`kube-proxy` 등)나 네트워크 관련 시스템 파드는 노드 네트워크를 직접 다뤄야 하므로 이 옵션을 씁니다.
* **확인:** 이 파드 안에서 `ifconfig`를 실행하면 컨테이너 가상 인터페이스가 아니라 **노드의 네트워크 어댑터들이 그대로 보입니다.**

---

## 13.1.2 호스트 포트에 바인딩하기 (`hostPort`)

노드 네트워크 전체가 아니라 **딱 하나의 포트만** 노드에 열고 싶을 때 사용합니다. `hostPort`를 지정하면 노드의 특정 포트로 들어온 트래픽이 **곧장 이 파드로 전달**됩니다.

```yaml
spec:
  containers:
  - image: luksa/kubia
    name: kubia
    ports:
    - containerPort: 8080
      hostPort: 9000        # 노드의 9000 포트 → 이 파드의 8080
```

### `hostPort` vs `NodePort` — 헷갈리지 말 것

| 구분 | `hostPort` | `NodePort` 서비스 |
|------|-----------|------------------|
| 연결 대상 | **파드가 떠 있는 그 노드**의 포트만 열림 | **모든 노드**의 같은 포트가 열림 |
| 트래픽 경로 | 노드 → 파드 (직통) | 어느 노드든 → 서비스 → (다른 노드의) 파드로 분산 |
| 파드 개수 | 노드당 1개만 가능 (포트 중복 불가) | 여러 파드로 로드밸런싱 |

* **비유:** `hostPort`는 **"이 건물 9000호로 직접 오세요"**, `NodePort`는 **"우리 체인점 어느 지점에 가도 9000번 창구로 안내됩니다"**.
* **주의:** 한 노드에서 같은 `hostPort`를 두 파드가 동시에 쓸 수 없어, 노드당 파드가 1개로 제한됩니다. (DaemonSet 형태 시스템 파드에 잘 어울림)

## 13.1.3 노드의 PID·IPC 네임스페이스 사용하기 (`hostPID`, `hostIPC`)

* **`hostPID: true`** — 파드가 **노드에서 실행 중인 모든 프로세스를 자기 프로세스처럼** 볼 수 있습니다. 컨테이너 안에서 `ps aux`를 치면 노드 전체 프로세스가 줄줄이 나옵니다.
* **`hostIPC: true`** — 파드의 컨테이너가 **노드의 다른 프로세스와 IPC(프로세스 간 통신)** 를 주고받을 수 있습니다. 공유 메모리 등으로 노드의 프로세스와 직접 대화하는 것이죠.

```yaml
spec:
  hostPID: true
  hostIPC: true
  containers:
  - name: main
    image: alpine
    command: ["/bin/sleep", "999999"]
```

* **비유:** `hostPID`는 **건물 전체의 CCTV 관제실에 들어가 모든 세대의 움직임을 보는 것**, `hostIPC`는 **세대 간 벽에 구멍을 뚫어 옆집과 직접 물건을 주고받는 것**과 같습니다.
* **실제 사용처:** 노드 모니터링·디버깅 도구처럼 호스트 프로세스를 들여다봐야 하는 파드.

---

## 결론: "벽을 허무는 권한"은 최소한으로

호스트 네임스페이스 옵션들은 **파드 격리라는 쿠버네티스의 기본 안전장치를 일부러 깨는 행위**입니다. 편리하지만, 침해당했을 때 노드 전체가 위험에 빠질 수 있는 강력한 권한이죠.

따라서 이런 옵션은 **시스템 수준의 신뢰된 파드에만** 부여하고, 일반 애플리케이션 파드에는 절대 허용해선 안 됩니다. 다음 절(13.3)에서 배울 **PodSecurityPolicy / Pod Security Admission**이 바로 "어떤 파드가 이런 위험한 옵션을 쓸 수 있는지"를 통제하는 장치입니다.
