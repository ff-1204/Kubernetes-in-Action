# 13.3 파드의 보안 관련 기능 사용 제한 (PodSecurityPolicy)

## 들어가기 전에: "양심"이 아니라 "규칙"으로 막기

13.2에서 본 보안 컨텍스트는 **파드 작성자가 스스로 거는 자물쇠**였습니다. 문제는, 악의적이거나 부주의한 사용자가 `privileged: true`나 `hostNetwork: true`를 그냥 써버리면 **아무도 막을 수 없다**는 것이었죠.

**PodSecurityPolicy(PSP)** 는 이 문제를 클러스터 차원에서 해결합니다. PSP는 **"우리 클러스터에 들어오는 모든 파드가 반드시 지켜야 할 보안 규칙집"** 입니다. 규칙을 어기는 파드는 **API 서버 단계에서 생성 자체가 거부**됩니다.

* **비유:** 보안 컨텍스트가 "각자 알아서 문을 잠그세요"라면, PSP는 **건물 입구의 보안 검색대(Admission Controller)** 입니다. 위험한 물건(설정)을 가진 사람은 아예 입장이 안 됩니다.
* **동작 위치:** PSP는 **Admission Control 플러그인**으로 동작합니다. 파드 생성 요청이 들어오면, 인증·인가를 통과한 뒤 이 검색대에서 정책 검사를 받습니다.

> 🕒 **중요 — 시대 변화 (책 그대로 + 최신 보충)**
> 이 책은 PSP가 표준이던 시절(쿠버네티스 1.x 초기)에 쓰였습니다. 그러나 **PSP는 1.21에서 deprecated 되고, 1.25에서 완전히 제거**되었습니다. 현재 표준은 그 뒤를 잇는 **Pod Security Admission(PSA)** 입니다. (이 절 끝의 "최신 보충" 참고)
> → 개념(파드가 쓸 수 있는 위험 기능을 클러스터가 통제한다)은 동일하니, **PSP는 "이런 아이디어가 있었다"는 역사적 맥락**으로 이해하고, 실무는 PSA로 적용하면 됩니다.

## 13.3.1 PodSecurityPolicy 리소스 소개

PSP는 네임스페이스에 속하지 않는 **클러스터 수준 리소스**로, 파드가 쓸 수 있는/없는 보안 기능을 한데 정의합니다.

```yaml
apiVersion: policy/v1beta1
kind: PodSecurityPolicy
metadata:
  name: default
spec:
  hostIPC: false
  hostPID: false
  hostNetwork: false           # 13.1의 호스트 네임스페이스 사용 금지
  hostPorts:
  - min: 10000
    max: 11000                 # 허용할 hostPort 범위 제한
  privileged: false            # 13.2의 특권 모드 금지
  readOnlyRootFilesystem: true # 루트 FS 읽기 전용 강제
  runAsUser:
    rule: RunAsAny
  fsGroup:
    rule: RunAsAny
  supplementalGroups:
    rule: RunAsAny
  seLinux:
    rule: RunAsAny
  volumes:
  - '*'                        # 모든 볼륨 타입 허용
```

* 이 정책 하나로 **특권 모드·호스트 네임스페이스를 금지**하고, **hostPort 범위를 제한**하며, **읽기 전용 루트 파일시스템을 강제**할 수 있습니다.
* 규칙을 위반하는 파드 생성 요청은 `Error ... forbidden` 메시지와 함께 거부됩니다.

---

## 13.3.2 `runAsUser`, `fsGroup`, `supplementalGroups` 정책 이해

이 항목들은 단순 허용/금지가 아니라 **"규칙(rule)"** 으로 표현됩니다.

* **`RunAsAny`** — 아무 값이나 허용 (제한 없음)
* **`MustRunAs`** — 지정한 범위(range) 안의 값만 허용. 범위를 벗어나면 거부.

```yaml
  runAsUser:
    rule: MustRunAs
    ranges:
    - min: 2          # UID는 2~2 사이만 허용 → 사실상 UID 2로 강제
      max: 2
  fsGroup:
    rule: MustRunAs
    ranges:
    - min: 2
      max: 10
    - min: 20
      max: 30
```

* **비유:** `RunAsAny`는 "아무 자리에나 앉으세요", `MustRunAs`는 **"지정 좌석(2번)에만 앉으세요"**. 다른 자리에 앉으려 하면 입장 거부.

## 13.3.3 허용·기본·금지 Capabilities 설정

13.2.4에서 본 커널 기능(capability)도 PSP로 통제합니다. 세 가지 항목이 있습니다.

```yaml
  allowedCapabilities:        # 파드가 추가(add)해도 되는 기능 목록
  - SYS_TIME
  defaultAddCapabilities:     # 모든 컨테이너에 자동으로 추가될 기능
  - CHOWN
  requiredDropCapabilities:   # 반드시 제거(drop)되어야 하는 기능
  - SYS_ADMIN
  - SYS_MODULE
```

* **`allowedCapabilities`** — "이 열쇠들까지는 가져도 좋다"는 화이트리스트.
* **`defaultAddCapabilities`** — 파드가 따로 지정 안 해도 기본으로 끼워주는 기능.
* **`requiredDropCapabilities`** — "이 위험한 기능은 무조건 빼라" (위반 시 거부 또는 강제 제거).

---

## 13.3.4 사용할 수 있는 볼륨 타입 제한하기

파드가 마운트할 수 있는 **볼륨 종류**를 제한합니다. 예를 들어 `hostPath`(노드 디스크 직접 마운트)는 위험하므로 막고 싶을 때 유용합니다.

```yaml
  volumes:
  - emptyDir
  - configMap
  - secret
  - persistentVolumeClaim    # 이 네 종류만 허용
  # hostPath 등은 목록에 없으므로 금지됨
```

* **비유:** "외부에서 가져올 수 있는 짐(볼륨)은 **기내 반입 허용 품목**뿐입니다. 위험물(hostPath)은 반입 금지."

---

## 13.3.5 사용자·그룹별로 다른 PSP 할당하기

PSP는 **그 자체로는 아무에게도 적용되지 않습니다.** RBAC(12장)을 통해 **"누가 어떤 PSP를 사용할 수 있는지"** 를 연결해야 비로소 효력이 생깁니다.

흐름은 다음과 같습니다.

1. **PSP 생성** — 예: 느슨한 `privileged` 정책과 엄격한 `restricted` 정책 두 개.
2. **ClusterRole 생성** — 특정 PSP를 `use`할 수 있는 권한을 정의.
   ```yaml
   apiVersion: rbac.authorization.k8s.io/v1
   kind: ClusterRole
   metadata:
     name: psp-restricted
   rules:
   - apiGroups: ["policy"]
     resources: ["podsecuritypolicies"]
     verbs: ["use"]
     resourceNames: ["restricted"]
   ```
3. **RoleBinding / ClusterRoleBinding** — 이 ClusterRole을 특정 사용자·그룹·ServiceAccount에 연결.

* **결과:** 관리자(admin)에게는 `privileged` 정책을, 일반 사용자에게는 `restricted` 정책을 매핑하면 — **같은 클러스터인데도 누가 파드를 만드느냐에 따라 허용 범위가 달라집니다.**
* **비유:** VIP 손님(admin)에게는 전 구역 출입 팔찌를, 일반 손님(user)에게는 일반석 팔찌를 채우는 것.

## 결론 + 🆕 최신 보충: PSP는 갔다, 이제는 PSA

PSP의 핵심 아이디어는 **"파드가 쓸 수 있는 위험한 보안 기능을, 파드 작성자의 양심이 아니라 클러스터 규칙으로 강제한다"** 였습니다. 이 사상은 지금도 유효합니다.

다만 PSP는 **설정이 너무 복잡하고(특히 RBAC 연동), 적용 여부를 예측하기 어렵다**는 비판을 받아 폐기되었습니다.

### 후계자: Pod Security Admission (PSA) — 쿠버네티스 1.25+ 표준

PSA는 PSP의 복잡함을 버리고 **네임스페이스에 라벨만 붙이면** 동작합니다.

* **3단계 표준 프로필** (직접 규칙을 짤 필요 없음)
  * **`privileged`** — 제한 없음 (시스템 파드용)
  * **`baseline`** — 알려진 권한 상승만 차단 (적당한 보안)
  * **`restricted`** — 강력하게 제한 (best practice 강제: non-root, capability drop 등)
* **3가지 모드**
  * **`enforce`** — 위반 파드 생성 **거부**
  * **`audit`** — 거부하진 않고 **감사 로그**에 기록
  * **`warn`** — 사용자에게 **경고 메시지**만 표시

```yaml
apiVersion: v1
kind: Namespace
metadata:
  name: my-app
  labels:
    pod-security.kubernetes.io/enforce: restricted   # 이 네임스페이스의 파드는 restricted 강제
    pod-security.kubernetes.io/warn: restricted
```

* **비유:** PSP가 **"모든 검색 규칙을 직접 손으로 짜야 하는 보안 검색대"** 였다면, PSA는 **"공항 보안 등급(상·중·하)을 라벨로 고르기만 하면 되는"** 방식입니다.

> **정리:** 이 절을 공부할 때 — **개념과 동기는 PSP로 이해**하고, **실무 적용은 PSA(네임스페이스 라벨)로 한다**고 기억하면 됩니다.
