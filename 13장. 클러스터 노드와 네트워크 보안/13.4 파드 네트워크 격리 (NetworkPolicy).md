# 13.4 파드 네트워크 격리 (NetworkPolicy)

## 들어가기 전에: 쿠버네티스의 기본 네트워크는 "벽이 없다"

쿠버네티스의 기본 네트워크 모델은 **"모든 파드는 서로 통신할 수 있다"** 입니다. 네임스페이스가 달라도, 용도가 전혀 달라도, 기본적으로는 **모든 파드가 다른 모든 파드에 자유롭게 접속**할 수 있죠.

* **비유:** 거대한 **칸막이 없는 사무실(open floor)** 과 같습니다. 누구나 누구의 자리에든 걸어가 말을 걸 수 있습니다. 편하지만, **악의적인 파드 하나가 침투하면 옆자리·뒷자리 DB까지 마음껏 접근**할 수 있다는 뜻이기도 합니다.

**NetworkPolicy** 는 이 뻥 뚫린 사무실에 **칸막이와 출입 규칙**을 세웁니다. "누가 누구에게 말 걸 수 있는지(인바운드)", "어디로 나갈 수 있는지(아웃바운드)"를 라벨 셀렉터로 지정하는 방화벽 규칙입니다.

> ⚠️ **전제 조건:** NetworkPolicy는 **CNI 네트워크 플러그인이 지원해야** 동작합니다. (Calico, Cilium, Weave Net 등) 정책을 지원하지 않는 플러그인을 쓰면 NetworkPolicy를 만들어도 **조용히 무시**됩니다.

---

## 13.4.1 네임스페이스에서 네트워크 격리 켜기

NetworkPolicy는 **`podSelector`로 "대상 파드"를 고르는 순간**, 그 파드는 정책의 통제를 받습니다. 빈 셀렉터(`podSelector: {}`)를 쓰면 **네임스페이스의 모든 파드**가 대상이 됩니다.

```yaml
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: deny-all
spec:
  podSelector: {}        # 네임스페이스 내 모든 파드에 적용
  policyTypes:
  - Ingress              # 인바운드(들어오는) 트래픽을 통제
  # ingress 규칙이 비어 있음 → 모든 인바운드 차단
```

* **핵심 원리:** 어떤 파드가 **하나라도 NetworkPolicy의 대상이 되면**, 그 순간부터 그 파드는 **"명시적으로 허용된 트래픽 외에는 전부 차단(default deny)"** 으로 바뀝니다.
* 위 예시는 "허용 규칙이 하나도 없는 인그레스 정책"이라, 결과적으로 **모든 외부 접속을 막는 완전 격리** 상태가 됩니다.

---

## 13.4.2 특정 파드만 서버 파드에 접속 허용하기

이제 칸막이에 **"특정 명찰을 단 사람만 통과"** 라는 규칙을 답니다. 예: `app: webserver` 파드에는 **오직 `app: webserver-client` 라벨이 붙은 파드만** 접속 가능.

```yaml
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: postgres-netpolicy
spec:
  podSelector:
    matchLabels:
      app: database        # 보호 대상: DB 파드
  ingress:
  - from:
    - podSelector:
        matchLabels:
          app: webserver   # 이 라벨의 파드만 접속 허용
    ports:
    - port: 5432           # 그것도 5432 포트로만
```

* **비유:** "DB 금고실에는 **'webserver' 사원증을 가진 사람만**, 그것도 **5432번 문으로만** 들어올 수 있다." 다른 파드가 접근하면 연결이 차단됩니다.
* **효과:** 프론트엔드 파드가 침해당해도, **DB에 직접 붙을 수 있는 건 정해진 파드뿐**이라 피해 확산을 막습니다.

---

## 13.4.3 쿠버네티스 네임스페이스 간 격리하기

여러 팀이 네임스페이스로 나뉘어 있을 때, **다른 네임스페이스의 파드 접근을 막거나 허용**할 수 있습니다. 이때는 `namespaceSelector`를 씁니다.

```yaml
  ingress:
  - from:
    - namespaceSelector:
        matchLabels:
          tenant: manning    # tenant=manning 라벨이 붙은 네임스페이스에서만 허용
```

* **`podSelector`** 가 "같은 네임스페이스 안에서 누구냐"를 본다면, **`namespaceSelector`** 는 **"어느 동네(네임스페이스)에서 왔느냐"** 를 봅니다.
* **비유:** 사원증(파드 라벨)뿐 아니라 **소속 부서(네임스페이스 라벨)** 까지 확인하는 출입 통제. "마케팅 부서 건물에서 온 사람만 출입 허용."

---

## 13.4.4 CIDR 표기법으로 격리하기

파드 라벨이 아니라 **IP 대역(CIDR)** 으로 허용/차단할 수도 있습니다. 클러스터 외부의 특정 네트워크에서 오는 트래픽을 다룰 때 유용합니다.

```yaml
  ingress:
  - from:
    - ipBlock:
        cidr: 192.168.1.0/24    # 이 IP 대역에서 오는 트래픽만 허용
```

* **비유:** "라벨이고 뭐고, **이 우편번호(192.168.1.x) 지역에서 온 편지만** 받습니다."
* **사용처:** 클러스터 밖의 고정 IP 시스템(레거시 서버, 특정 게이트웨이 등)을 허용 목록에 넣을 때.

## 13.4.5 파드의 아웃바운드 트래픽 제한하기 (Egress)

지금까지는 "들어오는(Ingress)" 트래픽이었습니다. 반대로 **파드가 어디로 나갈 수 있는지(Egress)** 도 제한할 수 있습니다.

```yaml
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: egress-netpolicy
spec:
  podSelector:
    matchLabels:
      app: webserver
  policyTypes:
  - Egress
  egress:
  - to:
    - podSelector:
        matchLabels:
          app: database    # 이 파드는 database 파드로만 나갈 수 있음
    ports:
    - port: 5432
```

* **비유:** 인바운드가 "누가 우리 집에 들어올 수 있나"라면, 이그레스는 **"우리 집 사람은 어디로 외출할 수 있나"** 입니다. webserver 파드는 **오직 DB에만 외출 허가**, 인터넷 등 다른 곳으론 못 나갑니다.
* **왜 중요한가:** 파드가 탈취당해 **외부 공격 서버로 데이터를 빼돌리거나(exfiltration)**, 봇넷에 가담하는 것을 막습니다. (보안에서 점점 더 강조되는 부분)

---

## 결론: 두 겹의 방어선이 완성되다

13장 전체를 돌아보면, 쿠버네티스 보안은 **"안"과 "사이"** 두 방향으로 작동합니다.

| 절 | 통제 대상 | 한 줄 요약 |
|----|----------|-----------|
| 13.1 | 호스트 네임스페이스 | 파드가 **노드 자원**에 얼마나 손대는가 |
| 13.2 | 보안 컨텍스트 | 컨테이너가 **무슨 권한**으로 실행되는가 |
| 13.3 | PSP / PSA | 위험한 설정을 **클러스터가 강제로 통제** |
| 13.4 | NetworkPolicy | 파드 **사이의 통신**을 누가 허용받는가 |

NetworkPolicy는 그중 **"파드와 파드 사이의 방화벽"** 입니다. 기본은 모두 열려 있는 사무실이지만, **podSelector·namespaceSelector·ipBlock**으로 칸막이를 세우고, **Ingress(들어옴)·Egress(나감)** 양방향을 모두 잠글 수 있습니다.

> 💡 **실무 권장:** 운영 클러스터에서는 먼저 **default-deny**(모두 차단) 정책을 깔고, 꼭 필요한 통신만 하나씩 화이트리스트로 여는 **"제로 트러스트(zero-trust)"** 접근이 가장 안전합니다.
