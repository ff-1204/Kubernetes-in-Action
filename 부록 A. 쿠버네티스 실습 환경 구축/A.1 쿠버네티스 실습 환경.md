## Minikube vs kind vs K3s 비교

> ★ = 본 가이드 기본 권장 도구 (kind)

| 구분 | Minikube | **kind** ★ | K3s (Rancher/SUSE) |
| :--- | :--- | :--- | :--- |
| **K8s 종류** | 표준 K8s | **표준 K8s** (kubeadm 기반) | K3s (경량 배포판) |
| **설계 철학** | 로컬 개발·학습 샌드박스 | **K8s 공식 컨포먼스 테스트** | 엣지(Edge)/IoT 및 운영 최적화 |
| **자원 권장** | 최소 2GB RAM | 1GB RAM+ | server: 1.5GB+, agent: 0.5GB+ |
| **실행 방식** | VM/Docker/bare-metal (driver별) | Docker / Podman / nerdctl 컨테이너 | systemd 단일 바이너리 |
| **시작 속도** | 수 분 | 수 십초 | 수 초 |
| **컨테이너 런타임** | driver별 (docker→dockerd, kvm→containerd) | containerd | containerd 내장 |
| **멀티노드** | 가능 (무거움) | **공식 지원** (`kind create cluster --config`) | 노드별 머신 필요 |
| **주요 장점** | 풍부한 빌트인 애드온 (Dashboard 등) | K8s 공식, 표준 동작 보장 | 낮은 유지비, **운영 도입 가능** |
| **한계** | 호스트 자원 점유 높음 | Docker 의존, 운영 비권장 | in-tree cloud provider 제거, 일부 alpha 미포함 |
| **대응 방안** | Docker 드라이버 + 애드온 비활성화 | 학습·CI 한정 사용 | 도입 전 API 호환성 검증 |
| **라이선스** | Apache 2.0 | Apache 2.0 | Apache 2.0 |

## 가상화 계층에 따른 성능 영향 분석

배포판 선택 시 가장 고려해야 할 요소는 **'가상화 계층(Virtualization Layer)'** 유무에 따른 호스트 시스템의 부하입니다.

### Minikube (가상 머신 드라이버 사용 시)
* **구조:** [하드웨어] → [호스트 OS] → **[하이퍼바이저] → [게스트 OS]** → [K8s]
* **특징:** Windows/macOS 환경에서 VirtualBox, VMware 등을 통해 실행됩니다.
* **영향:** **'이중 OS 실행'** 구조로 인해 실제 서비스 동작과 무관하게 가상화 계층 유지에만 상당량의 CPU와 RAM이 소모됩니다. 이로 인해 호스트 PC의 전체적인 응답 속도가 저하될 우려가 있습니다.

### kind (Docker 컨테이너 실행 시) ★
* **구조:** [하드웨어] → [호스트 OS] → **[Docker]** → [표준 K8s 노드 컨테이너] → [K8s]
* **특징:** **kubeadm 기반의 표준 K8s**를 Docker 컨테이너로 띄우는 K8s 공식 도구입니다. 노드 안에서 도는 것이 **표준 K8s**이므로 책의 표준 예제와 동작이 일치합니다.
* **영향:** Docker 컨테이너 한 겹이 추가되지만, **표준 K8s 동작이 보장**되므로 컨포먼스 테스트나 책의 표준 예제를 가장 충실히 재현하고 싶을 때 적합합니다.

### K3s (네이티브 실행 시)
* **구조:** [하드웨어] → [호스트 OS] → [K8s 프로세스]
* **특징:** 가상화 계층 없이 리눅스 **호스트 OS 위에서 직접 프로세스로 실행**됩니다.
* **영향:** 중간 단계(오버헤드)가 거의 없어 하드웨어 성능을 그대로 활용 가능하며, 사양이 낮은 저전력 엣지 기기나 로컬 환경에서 매우 가벼운 동작을 제공합니다.

## 실습 환경에 따른 권장사항

실습자의 기기 부하·라이선스 비용·학습 시나리오를 모두 고려한 의사결정 가이드라인입니다.

### 사양에 따른 환경 추천
* **RAM 16GB 이상** → **kind** (멀티노드/HA 시뮬레이션도 여유, 책 예제 100% 호환) ★
* **RAM 8~16GB** → **kind** (단일/소수 노드만 띄우면 충분히 동작)
* **RAM 8GB 이하** → **kind** (단일 노드)도 가능, 더 가벼움이 필요하면 **K3s 직접 설치**

### 상업용 사용 가능 여부

학습뿐 아니라 회사 업무에서 사용할 경우, 라이선스 비용 발생 여부는 다음과 같습니다.

| 도구 | 라이선스 | 상업용 무료 | 비고 |
| :--- | :--- | :--- | :--- |
| **kind** ★ | Apache 2.0 | ◎ 무제한 | K8s SIG 공식 (kubernetes-sigs/kind), 운영 환경 적용은 비권장 (개발/CI 한정) |
| **Minikube** | Apache 2.0 | ◎ 무제한 | 자유 사용·수정·재배포 가능 |
| **K3s** | Apache 2.0 | ◎ 무제한 | CNCF 졸업 프로젝트, 운영 환경 적용 가능 |
| **Docker Engine** (docker-ce) | Apache 2.0 | ◎ 무제한 | WSL/Linux에 직접 설치하는 패키지 |
| **Docker Desktop** | 독점 (Docker 사) | △ **조건부** | 직원 250명 미만 **AND** 연 매출 $10M 미만 회사만 무료 / 그 외 유료 구독 ($9~/월) |
| **WSL2** | MS 독점 (Windows 구성요소) | ◎ Windows 라이선스 내 무료 | - |

### 상업용 사용 시 권장 셋업
* **개인/학습/소규모 회사 (250명 미만)** → 어떤 조합이든 무료 사용 가능
* **중·대규모 회사 (250명 이상 또는 매출 $10M 이상)** → **Docker Desktop을 피해야 함**
  * **권장: WSL2 + Docker Engine(docker-ce) + kind** ★
  * 모든 구성요소가 Apache 2.0 라이선스로 회사 규모 무관 무료
* **운영 환경 도입** → **kubeadm / K3s / 클라우드 관리형(EKS·GKE·AKS)** (kind·Minikube는 운영 부적합)

### 종합 결정 매트릭스

| 사양 \ 클러스터 구성 | 단일 클러스터 | 멀티 클러스터/노드/HA |
| :--- | :--- | :--- |
| **RAM 16GB 이상** | **kind** | **kind** (`--config`로 멀티노드) |
| **RAM 8~16GB** | **kind** | **kind** (control-plane 3 + worker 2까지 가능) |
| **RAM 8GB 이하** | **kind** (단일 노드, 더 가벼움이 필요하면 K3s) | **kind** (control-plane 1 + worker 1~2 권장) |
