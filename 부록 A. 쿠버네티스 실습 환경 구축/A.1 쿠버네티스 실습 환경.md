> **K8s 공식 학습 도구**: [kubectl, kind, minikube, kubeadm](https://kubernetes.io/docs/tasks/tools/) 이 공식 권장됩니다. 본 가이드는 책 (『Kubernetes in Action』 2판)의 권장과 정렬하여 **kind (Kubernetes IN Docker)** 를 채택합니다. kubeadm 기반의 표준 K8s라 책 예제가 그대로 동작하며, 단일 노드부터 멀티노드·HA 시뮬레이션까지 도구 하나로 가능합니다.

## Minikube vs kind vs K3s vs k3d 비교

> **본 가이드 선택**: ★ **kind** — 책 2판이 사용하는 K8s 공식 도구. 표 안에서 ★로 표시됩니다.

| 구분 | Minikube | **kind** ★ | K3s (Rancher/SUSE) | k3d (K3s in Docker) |
| :--- | :--- | :--- | :--- | :--- |
| **K8s 종류** | 표준 K8s | **표준 K8s** (kubeadm 기반) | K3s (경량 배포판) | K3s |
| **설계 철학** | 로컬 개발·학습 샌드박스 | **K8s 공식 컨포먼스 테스트** | 엣지(Edge)/IoT 및 운영 최적화 | K3s를 Docker로 감싼 개발/CI 도구 |
| **자원 권장** | 최소 2GB RAM | 1GB RAM+ | server: 1.5GB+, agent: 0.5GB+ | K3s + Docker 오버헤드 (1GB+) |
| **실행 방식** | VM/Docker/bare-metal (driver별) | Docker / Podman / nerdctl 컨테이너 | systemd 단일 바이너리 | Docker 컨테이너 |
| **시작 속도** | 수 분 | 수 십초 | 수 초 | 수 초 |
| **컨테이너 런타임** | driver별 (docker→dockerd, kvm→containerd) | containerd | containerd 내장 | 노드 내 containerd |
| **멀티노드** | 가능 (무거움) | **공식 지원** (`kind create cluster --config`) | 노드별 머신 필요 | **`--servers N --agents M`** |
| **주요 장점** | 풍부한 빌트인 애드온 (Dashboard 등) | K8s 공식, 표준 동작 보장 | 낮은 유지비, **운영 도입 가능** | 멀티노드/멀티클러스터 즉석 구성 |
| **한계** | 호스트 자원 점유 높음 | Docker 의존, 운영 비권장 | in-tree cloud provider 제거, 일부 alpha 미포함 | Docker 의존, 운영 비권장 |
| **대응 방안** | Docker 드라이버 + 애드온 비활성화 | 학습·CI 한정 사용 | 도입 전 API 호환성 검증 | 학습·CI 한정, 운영은 K3s 직접 설치 |
| **라이선스** | Apache 2.0 | Apache 2.0 | Apache 2.0 | MIT |

## 가상화 계층에 따른 성능 영향 분석

배포판 선택 시 가장 고려해야 할 요소는 **'가상화 계층(Virtualization Layer)'** 유무에 따른 호스트 시스템의 부하입니다.

### Minikube (가상 머신 드라이버 사용 시)
* **구조:** [하드웨어] → [호스트 OS] → **[하이퍼바이저] → [게스트 OS]** → [K8s]
* **특징:** Windows/macOS 환경에서 VirtualBox, VMware 등을 통해 실행됩니다.
* **영향:** **'이중 OS 실행'** 구조로 인해 실제 서비스 동작과 무관하게 가상화 계층 유지에만 상당량의 CPU와 RAM이 소모됩니다. 이로 인해 호스트 PC의 전체적인 응답 속도가 저하될 우려가 있습니다.

### K3s (네이티브 실행 시)
* **구조:** [하드웨어] → [호스트 OS] → [K8s 프로세스]
* **특징:** 가상화 계층 없이 리눅스 **호스트 OS 위에서 직접 프로세스로 실행**됩니다.
* **영향:** 중간 단계(오버헤드)가 거의 없어 하드웨어 성능을 그대로 활용 가능하며, 사양이 낮은 저전력 엣지 기기나 로컬 환경에서 매우 가벼운 동작을 제공합니다.

### kind (Docker 컨테이너 실행 시)
* **구조:** [하드웨어] → [호스트 OS] → **[Docker]** → [표준 K8s 노드 컨테이너] → [K8s]
* **특징:** **kubeadm 기반의 표준 K8s**를 Docker 컨테이너로 띄우는 K8s 공식 도구입니다. 노드 안에서 도는 것이 K3s가 아닌 **풀 표준 K8s**라는 점이 k3d와의 핵심 차이입니다.
* **영향:** k3d와 동일하게 컨테이너 한 겹이 추가되지만, **표준 K8s 동작이 보장**되므로 컨포먼스 테스트나 책의 표준 예제를 가장 충실히 재현하고 싶을 때 적합합니다.

### k3d (Docker 컨테이너 실행 시)
* **구조:** [하드웨어] → [호스트 OS] → **[Docker]** → [K3s 노드 컨테이너] → [K8s]
* **특징:** K3s를 **Docker 컨테이너 단위로 띄우는 래퍼(wrapper)** 도구입니다. 한 호스트에서 여러 노드/클러스터를 격리하여 띄울 수 있습니다.
* **영향:** VM 같은 이중 OS 부하는 없지만 Docker 컨테이너 한 겹이 추가됩니다. 노드들이 모두 같은 호스트 Docker 데몬에 의존하므로 **진정한 HA는 불가능**하며, 학습·CI 환경에 적합합니다.

## 실습 환경에 따른 권장사항

실습자의 기기 부하·라이선스 비용·학습 시나리오를 모두 고려한 의사결정 가이드라인입니다.

### VM 회피 (WSL2 또는 Linux 활용)
* **문제:** Windows에서 VirtualBox나 VMware를 쓰는 것은 저사양 환경에 큰 부담이 됩니다.
* **대응:** Windows 사용자라면 **WSL2(Windows Subsystem for Linux)** 내에 Docker Engine + kind를 설치하는 것을 권장합니다. 가상 머신(VM)보다 가볍고 호스트 자원을 유연하게 공유합니다.

### 사양에 따른 환경 추천
* **RAM 16GB 이상** → **kind** (멀티노드/HA 시뮬레이션도 여유, 책 예제 100% 호환) — **본 가이드 기본**
* **RAM 8~16GB** → **kind** (단일/소수 노드만 띄우면 충분히 동작)
* **RAM 8GB 이하** → **K3s 직접 설치** (kind도 가능하지만 단일 노드 컨테이너 오버헤드가 부담된다면 K3s가 더 가벼움)

### 멀티 클러스터 유무에 따른 추천
* **단일 클러스터만 사용** → **kind** (`kind create cluster --name study` 한 줄, 본 가이드 기본)
* **멀티노드 단일 클러스터** (control-plane 1 + worker N) → **kind** (`kind create cluster --config kind-multi.yaml`)
* **HA 클러스터 시뮬레이션** (control-plane N) → **kind** (config에서 `role: control-plane`을 3개 선언, etcd Raft 합의 동작 학습 가능 — 단 진정한 HA는 물리 노드 분산 필요)
* **다중 독립 클러스터** (dev/staging/prod 분리) → **kind** (각 클러스터마다 별도 Docker 네트워크와 노드 컨테이너로 격리)
* **운영 환경 적용** → **kubeadm**(자체 운영) / **K3s 직접 설치**(엣지·소규모) / **GKE·EKS·AKS**(클라우드 관리형) — kind·k3d·Minikube는 학습/CI 전용

### 상업용 사용 가능 여부

학습뿐 아니라 회사 업무에서 사용할 경우, 라이선스 비용 발생 여부는 다음과 같습니다.

| 도구 | 라이선스 | 상업용 무료 | 비고 |
| :--- | :--- | :--- | :--- |
| **kind** ★ | Apache 2.0 | ◎ 무제한 | **본 가이드 기본**, K8s SIG 공식 (kubernetes-sigs/kind), 운영 환경 적용은 비권장 (개발/CI 한정) |
| **Minikube** | Apache 2.0 | ◎ 무제한 | 자유 사용·수정·재배포 가능 |
| **K3s** | Apache 2.0 | ◎ 무제한 | CNCF 졸업 프로젝트, 운영 환경 적용 가능 |
| **k3d** | MIT | ◎ 무제한 | 단, **운영 환경 적용은 비권장** (개발/CI 한정) |
| **Docker Engine** (docker-ce) | Apache 2.0 | ◎ 무제한 | WSL/Linux에 직접 설치하는 패키지 |
| **Docker Desktop** | 독점 (Docker 사) | △ **조건부** | 직원 250명 미만 **AND** 연 매출 $10M 미만 회사만 무료 / 그 외 유료 구독 ($9~/월) |
| **WSL2** | MS 독점 (Windows 구성요소) | ◎ Windows 라이선스 내 무료 | - |

### 상업용 사용 시 권장 셋업
* **개인/학습/소규모 회사 (250명 미만)** → 어떤 조합이든 무료 사용 가능
* **중·대규모 회사 (250명 이상 또는 매출 $10M 이상)** → **Docker Desktop을 피해야 함**
  * **권장: WSL2 + Docker Engine(docker-ce) + kind** ★
  * 모든 구성요소가 Apache 2.0 라이선스로 회사 규모 무관 무료
* **운영 환경 도입** → **kubeadm / K3s / 클라우드 관리형(EKS·GKE·AKS)** (kind·k3d·Minikube는 운영 부적합)

### 종합 결정 매트릭스

| 사양 \ 클러스터 구성 | 단일 클러스터 | 멀티 클러스터/노드/HA |
| :--- | :--- | :--- |
| **RAM 16GB 이상** | **kind** ★ | **kind** (`--config`로 멀티노드) ★ |
| **RAM 8~16GB** | **kind** ★ | **kind** (control-plane 3 + worker 2까지 가능) |
| **RAM 8GB 이하** | **kind** (단일 노드) 또는 K3s 직접 설치 (가장 가벼움) | **kind** (control-plane 1 + worker 1~2 권장) |

> **팁: kind 단일 도구 셋업 (라이선스 안전)**
> WSL2 + Docker Engine + kind 조합 하나로 단일 노드부터 멀티노드, 다중 클러스터, HA 시뮬레이션까지 모두 가능합니다. **Docker Desktop이 아닌 Docker Engine(docker-ce)** 을 WSL에 직접 설치하면 회사 규모와 관계없이 모든 구성요소가 무료(Apache 2.0)로 동작합니다.

### 『쿠버네티스 인 액션』 학습용 추천 환경

이 책(『Kubernetes in Action』, Marko Lukša 저)은 다음 환경을 공식 권장합니다.

| 책의 권장 환경 | 용도 | 본 가이드에서의 채택 |
| :--- | :--- | :--- |
| **Minikube** | 로컬 단일 노드 학습 (1판 주력) | (선택지로 유지, 본 가이드는 kind 위주) |
| **kind** (2판부터) | Docker 기반 로컬 멀티 노드 | **★ 본 가이드 기본** |
| **Docker Desktop 빌트인 K8s** | macOS/Windows 사용자 간편 셋업 | (라이선스 이슈로 비추천) |
| **GKE / EKS** | 클라우드 멀티 노드 실습 | (선택 사항, 비용 발생) |

본 학습 가이드는 책 2판의 **kind**를 그대로 채택합니다. 이유는 다음과 같습니다.

* **책과 정렬**: 책 2판이 kind를 사용하므로 예제 환경 차이로 인한 마찰 최소
* **표준 K8s 보장**: kubeadm 기반의 풀 표준 K8s — 책의 모든 `kubectl` 명령과 매니페스트가 그대로 동작
* **단일 도구로 통합**: 단일 노드/멀티노드/다중 클러스터/HA 시뮬레이션 모두 kind 하나로 가능
* **K8s 공식**: kubernetes-sigs 공식 도구로 회사 환경에서도 정당화 명확

**책 예제와의 호환성 (kind 환경)**:

| 항목 | 책 (kind 2판 기준) | 본 가이드 환경 |
| :--- | :--- | :--- |
| `kubectl` 명령 | 표준 | ◎ 100% 동일 |
| Ingress 컨트롤러 | nginx-ingress 별도 설치 | ◎ 책과 동일 (kind 전용 매니페스트로 설치) |
| LoadBalancer 타입 | extraPortMappings로 노출 | ◎ 책과 동일 |
| 컨테이너 런타임 | containerd (kind 노드 내) | ◎ 책과 동일 |
| 멀티 노드 등장 챕터 | `kind create cluster --config` | ◎ 책과 동일 |

> **결론:** 이 책으로 학습한다면 **WSL2 + Docker Engine + kind 조합**이 책의 의도를 가장 충실하게 따르면서도 노트북 자원·라이선스 비용·실습 안정성을 모두 만족시키는 최적 선택입니다.

## 운영 환경(Production) 옵션

학습 단계 이후 실제 운영에 도입할 때의 표준 옵션입니다. [K8s 공식 setup 가이드](https://kubernetes.io/docs/setup/) 와 정렬됩니다.

### 자체 운영 (Self-Managed)

| 도구 | 특징 | 적합한 규모 |
| :--- | :--- | :--- |
| **kubeadm** | K8s 공식 운영 부트스트랩. HA 컨트롤 플레인·etcd 클러스터 직접 구성 | 중·대규모, 자체 데이터센터 |
| **K3s** | embedded etcd로 HA 가능, 단일 바이너리로 가벼움 | 엣지/IoT, 소규모 온프레 |
| **kubespray** | Ansible 기반 자동화 | 다중 노드 자동 프로비저닝 |

> **운영급 HA의 원리(왜 마스터 3 + 워커 2인가)**는 [A.4 쿠버네티스 최소 이중화(HA) 구성](A.4%20쿠버네티스%20최소%20이중화(HA)%20구성.md) 에서 etcd Raft 합의와 워크로드 가용성 측면으로 설명합니다.

### 클라우드 관리형 (Managed Services)

컨트롤 플레인을 사업자가 운영해주는 방식. **사용자는 워커 노드와 워크로드만 신경 쓰면 됩니다.**

| 사업자 | 서비스 | 특징 |
| :--- | :--- | :--- |
| AWS | **EKS** | 시장 점유율 1위, IAM 통합 |
| Google Cloud | **GKE** | K8s 원조(Google), Autopilot 모드 |
| Azure | **AKS** | MS 생태계(AD, DevOps) 연계 |
| Naver Cloud | **NKS** | 국내 컴플라이언스 |

* **장점**: etcd/마스터 장애를 사업자가 책임짐, 자동 업그레이드
* **단점**: 컨트롤 플레인 비용(~$73/월부터), 벤더 종속

### 엔터프라이즈 배포판

| 제품 | 제공사 | 포지셔닝 |
| :--- | :--- | :--- |
| **OpenShift** | Red Hat | 빌트인 CI/CD·보안·라우터 |
| **Rancher** | SUSE | **K3s 만든 곳**, 멀티 클러스터 관리 GUI |
| **Tanzu** | VMware | vSphere 통합 |

> **본 가이드와의 연결**: 본 가이드(kind)에서 익힌 모든 `kubectl` 명령과 YAML 매니페스트는 위 모든 옵션에서 그대로 동작합니다. kind는 kubeadm 기반의 표준 K8s이므로 차이는 인증(IAM/OIDC), CSI/CNI 드라이버, 부가 기능(Routes, ALB 등) 정도이며, 표준 K8s API 부분은 100% 호환됩니다.
