# A.4 쿠버네티스 최소 HA 구성 (마스터 3 + 워커 2)

쿠버네티스 클러스터는 **컨트롤 플레인(Control Plane)** 이 죽으면 새 Pod 스케줄링이 멈추고, etcd가 깨지면 클러스터 상태 자체가 손상됩니다. 워커 노드 한 대만 두고 운영하면 그 노드가 죽는 순간 모든 앱이 `Pending`으로 멈춰버립니다. 그래서 운영 환경에서는 마스터·워커 **양쪽 다 이중·삼중으로 보호하는 HA 구성**이 필수입니다.

이 문서는 **"운영 환경에서 최소 몇 대로 HA를 구성해야 하는가?"** 라는 질문에 답합니다. 결론부터 말하면 **마스터 3대 + 워커 2대**가 운영급 최소 구성입니다. 마스터 3대는 *합의 알고리즘* 측면에서, 워커 2대는 *워크로드 가용성* 측면에서 각각 "한 대 죽어도 살아있는" 최소선이에요. 분산 시스템 원리로 그 이유를 차례차례 보겠습니다.

---

## 0. 컨트롤 플레인이란?

쿠버네티스 클러스터의 **'두뇌' 역할을 하는 노드(들)** 입니다. 클러스터에서 일어나는 모든 결정을 내리고 그 상태를 기록하는 곳이에요. [1.3 쿠버네티스 소개](../1장.%20쿠버네티스%20소개/1.3%20쿠버네티스%20소개.md)에서 썼던 **마라탕 프랜차이즈 본사** 비유를 이어가면 이렇습니다.

| 구분 | 비유 | 역할 |
| :--- | :--- | :--- |
| **컨트롤 플레인** | **프랜차이즈 본사 사무실** | "어느 매장에 밀키트 몇 개 보낼지" 결정·지시·기록 |
| **워커 노드** | **각 지점 매장** | 본사 지시대로 실제 마라탕(Pod)을 끓이는 곳 |

사용자가 `kubectl apply -f deployment.yaml`을 실행하면 → **컨트롤 플레인**이 요청을 받아서 → "어느 워커 노드에 띄울까?" 결정하고 → 해당 워커 노드에게 "이거 실행해" 명령을 내립니다.

### 컨트롤 플레인의 4대 컴포넌트

| 컴포넌트 | 비유 | 역할 |
| :--- | :--- | :--- |
| **kube-apiserver** | **본사 안내데스크** | 모든 요청이 들어오는 유일한 정문. `kubectl`도 여기로 통신 |
| **etcd** | **본사 금고/장부** | 클러스터의 모든 상태를 저장하는 분산 DB (Raft 합의) |
| **kube-scheduler** | **배차 담당자** | 새 Pod를 **어느 워커 노드**에 띄울지 결정 |
| **kube-controller-manager** | **현장 감독관** | "복제본 3개" 같은 선언을 실시간 감시·보정 (self-healing) |

### 워커 노드의 컴포넌트 (참고)

| 컴포넌트 | 역할 |
| :--- | :--- |
| **kubelet** | 각 워커 노드의 '지점장'. 컨트롤 플레인 명령을 받아 컨테이너 실행 |
| **kube-proxy** | 네트워크 라우팅 담당 |
| **컨테이너 런타임** | 실제로 컨테이너를 띄우는 엔진 (containerd 등) |

> **핵심:** 컨트롤 플레인 = '본사'이기 때문에 죽으면 클러스터 전체가 마비됩니다. 반대로 워커 노드는 '매장'이라 한 곳 문 닫혀도 다른 매장에서 계속 운영됩니다. 그래서 **HA 논의의 핵심은 언제나 컨트롤 플레인**입니다.

---

## 1. 왜 HA가 필요한가

쿠버네티스는 **선언적 모델**로 동작합니다. "이 앱의 복제본을 3개 유지하라"는 사용자의 선언을 컨트롤 플레인이 실시간으로 감시하고 보정합니다. 그런데 그 컨트롤 플레인 자체가 죽으면?

* **새 Pod 스케줄링 불가:** 죽은 노드 위에 있던 앱이 다른 노드로 옮겨가지 못합니다.
* **클러스터 상태 변경 불가:** `kubectl apply` 같은 모든 명령이 거부됩니다.
* **etcd 손상 시 복구 불가:** 클러스터 전체 상태를 잃을 위험이 있습니다.

즉, **컨트롤 플레인이 단일 장애점(SPOF, Single Point Of Failure)이면 쿠버네티스를 쓰는 의미 자체가 흔들립니다.** HA 구성은 이를 방지하기 위한 필수 안전장치입니다.

---

## 2. 운영급 최소 구성: 마스터 3 + 워커 2

| 노드 | 개수 | 보호 대상 | 이유 |
| :--- | :--- | :--- | :--- |
| **마스터 (컨트롤 플레인)** | **3대 (홀수)** | 클러스터의 *의사결정* 능력 | etcd Raft quorum 유지, 1대 장애 허용 |
| **워커** | **2대 이상** | 사용자 *워크로드* (Pod) | 1대 장애 시 다른 노드로 Pod 재스케줄링 |
| **etcd** | (마스터에 임베디드) | 클러스터 상태 저장소 | 마스터 3대와 함께 3중화 |

3+2는 **운영급 최소선**입니다. 더 나아가면:

- 트래픽이 늘어나면 → **워커를 자유롭게 추가** (3, 5, 10, 100...)
- 더 강한 HA가 필요하면 → **마스터를 5, 7대로** 증설 (단, 항상 홀수)

다음 절들에서 "왜 마스터는 3대 홀수인가", "왜 1대로는 안 되는가", "왜 워커는 2대 이상인가"를 차례로 봅니다.

---

## 3. 왜 컨트롤 플레인이 홀수인가? — '방장 선거' 비유

답은 컨트롤 플레인의 두뇌인 **etcd**가 사용하는 **Raft 합의 알고리즘** 때문입니다.

### 가. Raft는 '방장 선거'이다

분산 데이터베이스는 여러 노드에 같은 데이터를 복제해서 안정성을 높이는데, 이때 **누가 쓰기를 주도할 것인가**를 정해야 합니다. 모두가 동시에 쓰면 데이터가 엉키니까요. Raft는 이 문제를 **리더 선출(Leader Election)** 로 해결합니다.

* 노드들이 투표해서 **단 하나의 리더**를 뽑습니다.
* 모든 쓰기 요청은 리더가 받아서 다른 노드에 전파합니다.
* 리더가 죽으면 남은 노드들이 다시 투표해서 새 리더를 뽑습니다.

리더 선출이 성립하려면 **과반수(Quorum)** 의 찬성표가 필요합니다.

### 나. 짝수 노드의 비극: "표가 갈리면 아무도 못 뽑는다"

> 6명이 모인 회의실에서 방장 선거를 한다고 쳐봅시다. 만약 표가 **3:3으로 갈리면** 방장은 누구도 되지 못합니다. 결국 회의가 멈춥니다.

etcd도 똑같습니다. 짝수 노드면 **Split-vote(표 갈림)** 가능성이 생겨 리더가 뽑히지 않을 수 있고, 그 순간 클러스터의 모든 쓰기가 멈춥니다.

### 다. 홀수의 마법: "과반수가 항상 결정된다"

홀수 노드는 표가 정확히 반으로 갈릴 수 없으므로 **항상 과반수가 결정됩니다.**

| 노드 수 | Quorum (과반수) | 허용 가능한 장애 노드 수 | HA 효율 |
| :--- | :--- | :--- | :--- |
| 1 | 1 | 0 | ❌ HA 아님 |
| 2 | 2 | 0 | ❌ 1대 죽으면 멈춤 |
| **3** | **2** | **1** | ✅ **최소 HA** |
| 4 | 3 | 1 | ⚠️ 3대보다 나을 게 없음 |
| **5** | **3** | **2** | ✅ 더 강한 HA |
| 6 | 4 | 2 | ⚠️ 5대보다 나을 게 없음 |
| **7** | **4** | **3** | ✅ 매우 강한 HA |

흥미로운 점은 **4대 구성이 3대보다 결코 낫지 않다**는 것입니다. 4대의 Quorum도 결국 3이라 1대만 장애를 허용하는데, 노드 1대 더 들이는 비용 대비 가용성 이득이 없기 때문입니다. **노드를 늘릴 거면 5, 7로 늘려야 합니다.**

## 4. 왜 1대로는 안 되나? — Fault Tolerance

"단일 노드라도 etcd 데이터를 볼륨에 안전하게 저장하면 되지 않을까?" 라는 의문이 들 수 있습니다. 하지만 이는 **저장소 가용성과 합의 가용성을 혼동**한 것입니다.

| 구분 | 의미 | 해결 수단 |
| :--- | :--- | :--- |
| **저장소 가용성** | 데이터가 디스크에 안전하게 남아있는가? | 볼륨, 백업, 스냅샷 |
| **합의 가용성** | 지금 이 순간 쓰기 요청을 받을 수 있는가? | **노드가 살아있어야 가능** |

단일 노드가 죽는 순간 데이터는 디스크에 멀쩡히 있어도 **아무도 그 데이터에 접근하지 못합니다.** 클러스터의 모든 쓰기가 그 자리에서 멈춥니다.

**Raft의 안전 보장:** N개 노드의 클러스터는 **`(N-1)/2`개의 노드 장애를 허용**합니다.
* 1대 → 0대 장애 허용 ❌
* 3대 → 1대 장애 허용 ✓ (**최소 HA**)
* 5대 → 2대 장애 허용 ✓ (강한 HA)

---

## 5. 워커는 왜 2대 이상인가? — 워크로드 가용성

여기까지는 마스터(컨트롤 플레인) 이야기였습니다. 워커는 **etcd quorum에 참여하지 않으므로 합의 알고리즘 측면에선 자유**예요. 그럼에도 운영 환경에서 워커를 **최소 2대 이상** 두는 이유는 따로 있습니다.

### 가. 단일 워커의 함정: "Pod이 갈 곳이 없다"

워커 1대 위에 ReplicaSet으로 복제본 3개를 띄웠다고 칩시다.

```
[워커 1] : Pod-A, Pod-B, Pod-C   ← 모두 같은 노드
```

이 워커가 죽으면 컨트롤 플레인은 즉시 알아채고 "Pod 3개를 다시 띄워라" 명령을 내립니다. 그런데 **띄울 다른 노드가 없습니다.** 결국 모든 앱이 `Pending` 상태로 멈춰버려요. 마스터가 3대로 멀쩡하게 살아있어도 소용없습니다 — 일할 사람이 한 명 뿐이었으니까요.

### 나. 2대 이상이면 재스케줄링이 가능

```
[워커 1] : Pod-A, Pod-B
[워커 2] : Pod-C
```

워커 1이 죽으면 컨트롤 플레인이 워커 2로 Pod-A, Pod-B를 옮깁니다.

```
[워커 1] : ❌ down
[워커 2] : Pod-A, Pod-B, Pod-C   ← 자동 재배치
```

서비스가 잠깐 흔들릴 수는 있어도 **완전히 멈추진 않습니다.**

### 다. 마스터 HA vs 워커 HA — 같은 듯 다른 보호 장치

| 구분 | 마스터 (컨트롤 플레인) | 워커 |
| :--- | :--- | :--- |
| **보호 대상** | 클러스터의 *의사결정* 능력 | 사용자 *워크로드* (Pod) |
| **장애 시 영향** | "지시할 사람이 없다" → 새 배포·복구 불가 | "일할 사람이 없다" → 앱 다운 |
| **메커니즘** | etcd Raft 합의 (홀수 필수) | 단순 가용성 (개수 자유) |
| **운영급 최소** | **3대 (홀수)** | **2대 이상** |

마스터는 *합의*가 깨지면 안 되므로 홀수, 워커는 *대체 가능성*만 있으면 되므로 2대 이상입니다.

## 6. 쿠버네티스의 특수성: 컨트롤 플레인과 워커 플레인의 분리

다른 분산 시스템(예: Cassandra, Kafka)은 보통 **컨트롤에 참여하는 노드가 워커 역할도 같이 수행**합니다. 그래서 흔히 "노드 3개로 시작하세요"가 곧 전체 클러스터를 의미합니다.

쿠버네티스는 다릅니다.

| 구분 | 다른 분산 시스템 (Cassandra 등) | 쿠버네티스 |
| :--- | :--- | :--- |
| **노드 역할** | 컨트롤 + 워커가 한 노드에 통합 | **컨트롤 플레인 / 워커 플레인 분리** |
| **HA 단위** | 전체 노드 수 | **컨트롤 플레인 노드 수**만 핵심 |
| **워커 추가** | HA 구성에도 영향 | HA와 무관하게 자유 |

이 분리 덕분에 쿠버네티스는 **"마스터는 3대(또는 5, 7대)로 합의 단위를 안정시키고, 워커는 트래픽에 따라 자유롭게 2 → 100대까지 확장하는"** 운영이 가능합니다.

## 7. 직접 만들어보기 — kind / kubeadm / 클라우드 관리형

이론을 실제로 검증해보는 세 가지 경로입니다.

### 가. 학습용으로 빠르게 — kind (권장)

본 가이드의 기본 도구인 kind는 **단일 호스트의 Docker 컨테이너**로 마스터 3 + 워커 2 구성을 즉시 만들 수 있습니다. 진정한 HA는 아니지만 etcd Raft 합의와 워커 재스케줄링 동작을 그대로 관찰할 수 있어 학습용으로 최적입니다.

```bash
cat > kind-ha.yaml <<EOF
kind: Cluster
apiVersion: kind.x-k8s.io/v1alpha4
nodes:
- role: control-plane
- role: control-plane
- role: control-plane
- role: worker
- role: worker
EOF

kind create cluster --name ha --config kind-ha.yaml
```

검증:

```bash
kubectl get nodes --context kind-ha
# NAME                STATUS   ROLES           AGE   VERSION
# ha-control-plane    Ready    control-plane   3m    v1.x.x
# ha-control-plane2   Ready    control-plane   2m    v1.x.x
# ha-control-plane3   Ready    control-plane   2m    v1.x.x
# ha-worker           Ready    <none>          1m    v1.x.x
# ha-worker2          Ready    <none>          1m    v1.x.x
```

마스터 1대를 죽여보면서 [3절](#3-왜-컨트롤-플레인이-홀수인가--방장-선거-비유)·[4절](#4-왜-1대로는-안-되나--fault-tolerance)에서 본 Raft 합의를 직접 관찰할 수 있습니다.

```bash
docker stop ha-control-plane2
kubectl get nodes --context kind-ha    # NotReady, 그래도 클러스터는 동작 (quorum 유지)
docker stop ha-control-plane3          # 두 번째 마스터 다운 → quorum 깨짐 → 응답 없음
```

상세 절차는 [A.3 멀티노드 클러스터 구성(kind)](A.3%20멀티노드%20클러스터%20구성(kind).md) 의 3절을 참고하세요.

### 나. 진짜 운영용으로 — kubeadm

학습이 아닌 **실제 운영**에 도입할 때는 [kubeadm](https://kubernetes.io/docs/setup/production-environment/tools/kubeadm/high-availability/) 으로 물리 노드(또는 VM)를 분산하여 구성합니다. K8s 공식 부트스트랩 도구이며, 마스터 3대 + 워커 2대 이상을 별개 머신에 배치해야 진짜 HA가 됩니다.

핵심은 **stacked etcd vs external etcd** 토폴로지 선택입니다.

| 토폴로지 | 구조 | 적합한 경우 |
| :--- | :--- | :--- |
| **Stacked etcd** | etcd가 마스터 노드에 임베디드 | 머신 수 제한, 가장 일반적 |
| **External etcd** | etcd가 별도 머신에 분리 | etcd와 컨트롤 플레인 장애 격리, 더 높은 안정성 |

자세한 가이드: [kubeadm HA topology](https://kubernetes.io/docs/setup/production-environment/tools/kubeadm/ha-topology/)

### 다. 클라우드 관리형 — EKS / GKE / AKS

**컨트롤 플레인의 HA를 사업자가 책임**져 줍니다. 사용자는 워커 노드 풀과 워크로드만 관리하면 됩니다.

| 서비스 | HA 보장 |
| :--- | :--- |
| AWS **EKS** | 멀티 AZ 컨트롤 플레인 자동 (3 AZ에 etcd 분산) |
| Google **GKE** | regional 클러스터 선택 시 멀티 zone 마스터 |
| Azure **AKS** | uptime SLA 옵션으로 컨트롤 플레인 HA |

운영 환경에서 가장 빠른 HA 도입 경로입니다.

> 학습 환경에서는 단일 노드 kind로 대부분 충분합니다. **HA는 운영 환경에서의 안전망**이라는 점만 이해하고 넘어가도 되고, 직접 만져보고 싶으면 위 가-항(kind)으로 5분 안에 3+2 클러스터를 만들 수 있습니다.

## 요약 및 비유

| 개념 | 비유 | 기술적 의미 |
| :--- | :--- | :--- |
| **컨트롤 플레인 HA** | **방장 선거단 3명** | etcd Raft 클러스터 |
| **홀수 노드** | **표 갈림 방지** | Quorum 항상 결정 가능 |
| **과반수 합의** | **3명 중 2명 동의** | Raft Consensus |
| **워커 노드** | **방장 명령을 따르는 일꾼들** | **2대 이상 (워크로드 가용성)** |

---

## 결론

* **운영급 최소 HA = 마스터 3대(홀수) + 워커 2대 이상**
* **마스터 3대 이유:** etcd Raft의 과반수 합의 — 짝수면 표가 갈려 리더 선출에 실패하고, 1대면 fault tolerance가 0이 됩니다.
* **워커 2대 이유:** 1대 장애 시 Pod이 옮겨갈 다른 노드가 있어야 함 — 합의가 아닌 *대체 가능성*의 문제입니다.
* **쿠버네티스만의 장점:** 컨트롤 플레인과 워커 플레인이 분리되어, **마스터는 합의 단위(3, 5, 7…), 워커는 워크로드 단위(2 이상 자유)** 로 따로 설계할 수 있습니다.
